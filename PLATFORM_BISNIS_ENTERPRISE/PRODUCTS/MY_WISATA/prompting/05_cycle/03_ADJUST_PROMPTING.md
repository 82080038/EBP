## ADJUST PROMPTING WORKFLOW

```
1. IDENTIFY CHANGE
   - What is changing?
   - Why is it changing?
   - Who requested the change?

2. ANALYZE IMPACT
   - What is the impact?
   - What needs to be adjusted?
   - What can be reused?

3. CHOOSE STRATEGY
   - Requirement change
   - Priority change
   - Architecture change
   - Technology change
   - Scope expansion
   - Scope reduction
   - Feedback incorporation
   - Course correction

4. GENERATE ADJUSTED PROMPT
   - Use appropriate template
   - Include change context
   - Define adjustment plan
   - Set expectations

5. EXECUTE ADJUSTED PROMPT
   - Execute adjusted prompt
   - Monitor execution
   - Handle new changes

6. VALIDATE RESULT
   - Did adjustment work?
   - Is new direction correct?
   - Are requirements met?

7. UPDATE STATE
   - Update state file
   - Log adjustment
   - Update documentation
```

## ADJUST PROMPTING DECISION TREE

```
CHANGE REQUESTED
    ↓
WHAT TYPE OF CHANGE?
    ↓
├─ Requirement Change → Adjust Requirements → Re-prompt
├─ Priority Change → Reorder Queue → Re-prompt
├─ Architecture Change → Plan Migration → Re-prompt
├─ Technology Change → Plan Migration → Re-prompt
├─ Scope Expansion → Plan Phases → Re-prompt
├─ Scope Reduction → Clean Up → Re-prompt
├─ Feedback → Validate → Implement → Re-prompt
└─ Course Correction → Save State → New Direction → Re-prompt
```

## ADJUST PROMPTING BEST PRACTICES

1. **Always understand the change** - Don't adjust blindly
2. **Analyze the impact** - Understand what needs to change
3. **Choose appropriate strategy** - Use the right adjustment approach
4. **Plan the adjustment** - Don't jump into execution
5. **Save current state** - Always save before changing direction
6. **Validate the change** - Ensure the adjustment is correct
7. **Update documentation** - Keep documentation in sync
8. **Learn from adjustments** - Improve prompting over time

## ESCALATION CRITERIA

Escalate to human if:
- Adjustment cannot be determined
- Impact is too large to handle
- Change conflicts with critical requirements
- Architecture change is risky
- Technology change is not feasible
- Scope change is unreasonable
